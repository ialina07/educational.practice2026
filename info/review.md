## 1. Цель работы

Разработать консольное приложение на языке C, реализующее сжатие и восстановление файлов по алгоритму Хаффмана, а также экспериментально оценить эффективность алгоритма: измерить степень сжатия, время работы на данных разных типов и размеров, выполнить обработку результатов и построить графики.

## 2. Теоретическая часть

### 2.1. Основная идея алгоритма Хаффмана

Алгоритм Хаффмана - метод сжатия без потерь. Вместо фиксированной длины кода (например, 8 бит на символ в ASCII) он использует переменную длину: частым символам даются короткие коды, редким - длинные. За счёт этого средняя длина кода уменьшается.

### 2.2. Префиксные коды

Ключевое свойство алгоритма: **ни один код не является префиксом (началом) другого кода**. Это позволяет однозначно декодировать битовый поток без разделителей.

### 2.3. Термины в данной работе

**"Символ"** - это не только буква, но и любой байт (0-255) или даже набор байтов. Программа должна работать с произвольными бинарными файлами (изображения, исполняемые файлы, архивы), а не только с текстом. Алфавит в данной реализации - 256 возможных значений байта.

### 2.4. Построение дерева Хаффмана

Процесс построения дерева:

1. **Подсчёт частот** - первый проход по файлу: для каждого байта (0–255) подсчитывается, сколько раз он встретился.
2. **Создание очереди с приоритетами** - для каждого символа с непустой частотой создаётся узел-лист (содержит символ и его частоту). Узлы помещаются в очередь; приоритет выше у меньшей частоты.
3. **Склеивание** - пока в очереди больше одного узла:
   - Извлечь два узла с наименьшими частотами
   - Создать новый узел-родитель: его частота = сумма частот дочерних узлов; дочерние узлы становятся левым и правым потомком
   - Поместить родителя обратно в очередь
4. **Корень** - последний оставшийся узел становится корнем дерева Хаффмана.

В задании упомянут альтернативный способ (массив деревьев), но очередь с приоритетами за логарифмическое время предпочтительнее для производительности.

### 2.5. Генерация таблицы префиксных кодов

Дерево обходится от корня к листьям:
- При переходе на левого потомка к текущему коду дописывается `0`
- При переходе на правого потомка дописывается `1`

Когда достигнут лист (символ), накопленная битовая последовательность сохраняется как код этого символа. Хранить коды удобно в массиве из 256 элементов (индекс = символ).

### 2.6. Кодирование (сжатие)

Второй проход по исходному файлу:
- Каждый символ заменяется его кодом из таблицы
- Биты кода последовательно записываются в буфер (1 байт)
- Как только буфер заполняется до 8 бит, он записывается в выходной файл
- По окончании файла оставшиеся биты в буфере дописываются с выравниванием

**Важно:** перед сжатыми данными в файл обязательно записывается **заголовок** - информация, позволяющая при распаковке восстановить дерево. Обычно это массив частот символов (256 значений) или сама таблица кодов.

### 2.7. Декодирование (распаковка)

1. Из заголовка сжатого файла восстанавливаются частоты символов.
2. По частотам строится **точно такое же дерево Хаффмана** (алгоритм из п. 2.4).
3. Сжатые данные читаются побитово. Каждый бит задаёт направление обхода дерева: `0` - к левому потомку, `1` - к правому.
4. Когда обход достигает листа, символ из этого листа записывается в выходной файл.
5. Процесс повторяется, пока не будет восстановлено исходное количество байт.

### 3. Технические реализации алгоритма

#### 3.1. Реализации на C

- **huffman-archiver (SmnTin)** — производительная реализация с каноническими кодами Хаффмана. Пропускная способность: ~195 MB/s на сжатие, ~90 MB/s на распаковку (MacBook Pro M2 Pro).
- **C-Zip (ConnorDye)** — учебный проект с модульной архитектурой: отдельные модули для битовых операций, списков и деревьев.

#### 3.2. Реализации на других языках

- **C++**: - большинство реализаций используют STL: `priority_queue` для построения дерева, `unordered_map` для таблицы кодов.
- **Python**: реализации с библиотекой heapq и подробными примерами, а также **dahuffman** - Python-реализация с поддержкой потокового режима и предварительно обученных кодовых таблиц.

## 4. Ожидаемые результаты исследования

- **Текстовые файлы** (русский текст в UTF-8, английский текст в ASCII): степень сжатия ожидается 40–55% для английского и 50–65% для русского (из-за двухбайтового представления).

- **Бинарные файлы** (EXE, BMP): сжатие будет слабее - до 70–85%, так как байты распределены более равномерно.

- **Уже сжатые файлы** (ZIP, JPEG): алгоритм почти не уменьшит размер (95–100%), поскольку данные уже имеют высокую энтропию.

- **Распаковка быстрее сжатия**: при распаковке не нужно заново строить таблицу кодов и делать второй проход по данным - достаточно один раз восстановить дерево и обходить его по битам.

- **Очень маленькие файлы** (менее 1–2 КБ): сжатие может увеличить размер из-за служебного заголовка (1024 байта на хранение частот) и округления битов до байтов.

- **С ростом файла**: степень сжатия стабилизируется (вклад заголовка становится незначительным), время работы растёт линейно.

### 5. Критерии успешной работы

- Программа корректно восстанавливает исходный файл после сжатия.
- Получены измеримые результаты: степень сжатия и производительность (время сжатия и разжатия в секундах).
- Проведён анализ эффективности на различных типах данных: текстовые файлы, бинарные файлы, уже сжатые файлы (ZIP, JPEG).
- Выявлены сценарии, где алгоритм является эффективным, а где - нет.
